# CLAUDE.md

このリポジトリで作業する際のガイドです。

## プロジェクト概要

メンバーのタスクと負担を可視化するツール「タスク見える君」。
`tasks.html` をブラウザで開くだけで動作する。サーバー・ビルド・パッケージ管理・依存ライブラリは一切なし。

- 主要ファイル: [`tasks.html`](tasks.html) — HTML/CSS/JS をすべて内包した単一ファイル
- 仕様書: [`spec.md`](spec.md) — 機能・データモデル・画面構成の定義
- データ永続化: ブラウザの `localStorage`（キー: `task-tool-data-v1`）。外部送信は行わない。
- 共有・バックアップ: JSON エクスポート / インポート。

目的（詳細は [`spec.md`](spec.md) 参照）:

- メンバーが今どんなタスクを、いつからいつまで、どのくらいの負担で抱えているかの可視化
- 負担が高すぎる／低すぎるメンバーの早期発見（→ ヒアリングのトリガー）
- 個人のタスク管理
- 一人ひとりの負担の遷移の把握

設計思想（重要）: 負担値は実態を映す鏡。**負担が高いからといってタスクの負担値を下げる運用は行わない**。警告はヒアリング等の対話を促すシグナルとして扱う。

## 構成

すべて `tasks.html` 一枚に収める方針。

- `<style>` — 全UIのCSS（CSS変数でテーマ管理）
- `<script>` — 状態は単一オブジェクト `data` に保持し localStorage へ保存する
  - `data.members[]` … メンバー（`{id, name, capacityWarnHigh, capacityWarnLow, color}`、警告閾値デフォルトは上限80/下限60）
  - `data.tasks[]` … タスク（`{id, memberId, title, load, startDate, endDate, status, note}`）
  - `data.baseline` … 計画スナップショット（`{savedAt, tasks:[...]}` または `null`）。「予定 vs 実績」可視化用に単一・手動保存（上書き）。

### 負担モデル

- キャパシティ比率モデル。`load` は「そのメンバーのキャパシティ（基準100）の何%を占めるか」。
- タスクの負担は期間（`startDate`〜`endDate`）の各日に**均等**にかかるものとして扱う。
- ある日の**日次負担 = その日にアクティブなタスクの `load` 合計**。
- 合計が 100 を超えても**クランプせず**、超過を強調表示する（オーバーの可視化が目的）。
- 期間内に重い/軽い区間がある場合は、使用者がタスクを分割して入力する運用。

### 画面構成（ビュー）

- 時間軸は**週単位**（月曜始まり）。表示期間は**四半期(13週)/半年(26週)/1年(52週)**を切替（中長期の負担俯瞰が目的）。
- チームビュー … 全メンバーを縦に並べ、負担・タスク期間を俯瞰
- 個人ビュー … 特定メンバーに絞ってタスク管理
- ガント風チャート … 横軸=週・縦軸=メンバー、タスクを期間の帯で表示（現在週をハイライト）
- 負担推移グラフ … メンバーごとの週ごとの平均負担の折れ線（100ラインを補助線）。**計画ベースライン（`data.baseline`）をトグルで点線・淡色で重ね描き**し、予定と実績の乖離を可視化（参照線であり警告判定には影響しない）。

### 警告判定

- 判定単位は**期間平均**。対象期間は**現在表示中の期間**（表示期間切替・前後移動に連動）の週平均負担を算出する。
- 平均が `capacityWarnHigh`（デフォルト80）超 → 過負荷警告。`capacityWarnLow`（デフォルト60）未満 → 低負荷警告。
- 平均に加え、対象期間内の**最大（ピーク）週負荷も併記**し、ピークが上限超なら強調表示する。
- 警告が出ても負担値は自動変更しない（詳細は [`spec.md`](spec.md) 5.4章）。

## 動作確認

ビルド不要。`tasks.html` をブラウザで直接開くか、任意の静的サーバーで配信する。

```
npx --yes http-server -p 8777 -c-1
```

（`python` はこの環境ではWindowsストアのスタブで動かないため Node を使う。）

## 開発ワークフロー（必須ルール）

以下は**必ず守る**こと:

1. **作業は機能単位で Issue を立ててから始める。** 実装の前に GitHub Issue を作成し、何を作るか・受け入れ条件を明記する。
2. **作業はブランチを切って行い、PR を出す。** 直接 `main` にコミットしない。PR 本文に `Closes #<issue番号>` を入れて Issue と紐付ける。
3. **PR を出した後、必ず `/code-review` を実施する。** 指摘事項はマージ前に対応するか、対応しない場合は PR コメント or 別 Issue として残す。
4. **`/code-review` の指摘は次のサイクルで処理する。**
   1. 指摘内容を PR コメントとして残す（対応する／しないの方針も記載）。
   2. 対応すると判断した指摘を修正してコミット・プッシュする。
   3. 再度 `/code-review` を実施する。
   4. 新たな指摘がなくなるまで 1〜3 を繰り返す。問題がなければマージ確認に移る。

### 補足

- コミット・プッシュ・PR作成・マージは、ユーザーが明示的に指示したときのみ行う。
- コミットメッセージ・PR・Issue は日本語で記述する。
- 単一ファイル構成を崩さない（外部依存・ビルド工程を持ち込まない）ことを基本方針とする。
